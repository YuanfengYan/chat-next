# 认证与后台

## 1. Better Auth 配置

认证实例位于 `src/lib/auth/auth.server.ts`，使用 Prisma Adapter 和 PostgreSQL，基础路径为 `/api/auth`。启用：

- 邮箱密码注册与登录，密码长度 8..128。
- 注册后自动登录。
- 可选 GitHub OAuth；只有 Client ID 和 Secret 同时存在时启用。
- 数据库会话，有效期 7 天、1 天刷新周期、5 分钟 Cookie Cache。
- Better Auth 用户字段映射：`name -> displayName`、`image -> avatarUrl`。

客户端与服务端共用 Zod 输入规则；邮箱会去空格、转小写并校验格式。服务端 Hook 再次校验，禁用账号以统一凭证错误拒绝，避免泄露账号状态。

## 2. 会话保护

`getCurrentSession()` 使用当前请求 headers 调用 Better Auth。首页和聊天页未登录时跳转 `/auth`；聊天 Route Handler 独立执行鉴权并返回 401。页面保护不能替代 API 鉴权。

数据库创建会话前检查 `User.status`，`DISABLED` 用户不能建立新会话。

## 3. 管理员判定

满足任一条件即为管理员：

- 数据库角色 key 为 `admin` 或 `super_admin`。
- 用户邮箱出现在逗号分隔的 `ADMIN_EMAILS` 环境变量中。

`getAdminViewer()` 只查询展示和授权所需字段，输出 DTO，不把完整 User、Account 或凭证字段传给界面。

## 4. 后台模块

后台当前为服务端只读查询：

- 仪表盘：云端会话、活跃会话、工具、启用工具、注册用户及最近记录。
- 工具：工具目录、模型绑定权限、调用日志、HTTP ToolVersion 运行配置。
- 助手：以 `PromptTemplate` 作为稳定助手身份，展示版本、模型参数和工具绑定。
- 知识库：无数据库模型，服务返回空 DTO，界面显示建设中状态。

“MCP / Function 配置”页面当前实际读取 HTTP `ToolVersion`，并未实现 MCP 协议客户端或通用 Function 执行器。

## 5. 安全注意事项

- 后台后续新增写操作时，应在每个 Route Handler 或 Server Action 内再次调用权限校验。
- `ADMIN_EMAILS` 适合早期部署白名单；稳定生产环境宜以数据库 RBAC 为主。
- Account 中的 OAuth token、PlatformCredential、ToolCredential 不得进入 DTO、日志或客户端。
- Cookie、会话撤销、OAuth 回调安全由 Better Auth 配置和部署域名共同决定，生产环境必须设置强随机 `BETTER_AUTH_SECRET` 和准确的 `BETTER_AUTH_URL`。
