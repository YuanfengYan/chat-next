# 系统架构

## 1. 目录结构

```text
src/
├─ app/                    Next.js 路由、布局、参数解析和 HTTP 入口
├─ features/
│  ├─ chat/               聊天会话、消息、流式交互与持久化
│  ├─ ai/                 模型、Skill、Tool、Provider 与流协议
│  ├─ auth/               登录注册、会话读取与认证客户端
│  └─ admin/              后台 DTO、查询、操作和界面
├─ shared/                基础 UI、主题组件和无业务工具
├─ infrastructure/        Prisma 与凭证安全等技术实现
└─ generated/             Prisma 生成代码
tests/                    集中的单元、组件和集成测试
```

`app` 目录使用 `(frontend)` 和 `(admin)` 路由组组织布局，路由组名称不进入 URL。`page.tsx` 只解析参数并组合领域组件，`route.ts` 只委托领域 handler。

## 2. 领域内部边界

- `domain`：稳定类型、协议和纯函数，不依赖 React、Next.js、数据库或供应商 SDK。
- `application`：协调领域用例和客户端交互流程，例如 `useChatController`。
- `client`：浏览器组件、Store、HTTP Repository 和 Transport。
- `server`：鉴权后的用例入口、服务端业务、Provider、Tool 和持久化适配器。
- `components`：认证和后台等领域界面；不得直接读取环境变量或 Prisma。

不是每个领域都必须创建所有子目录。只有出现对应职责时才增加目录，避免空抽象。

## 3. 依赖规则

```text
app  ───────▶ features ───────▶ shared
                  │
                  └───────────▶ infrastructure   （仅 server 代码）
```

- `infrastructure` 和 `shared` 不得反向依赖任何 feature。
- feature 间只依赖对方公开的 domain 或 application 接口。
- client 文件不得导入 `server`、`infrastructure`、`generated` 或供应商 SDK。
- Prisma 实体只存在于服务端适配器内部，对 UI 返回稳定 DTO 或领域对象。
- `.server.ts` 模块必须保持 server-only，不得仅为复用类型被客户端导入；共享类型应提升到 domain。
- 不建立聚合 client/server 的根级 barrel，使用明确路径表达依赖方向。

## 4. 核心请求链路

聊天请求由 `app/api/chat/route.ts` 委托聊天 handler。handler 完成认证和用例协调；请求与图片校验位于聊天 validation；云端消息和 Generation 生命周期位于聊天 service；模型、Skill 与 Tool 执行位于 AI orchestration；数据库访问只通过服务端适配器和 infrastructure。

会话 API 同样只从 Route Handler 转发到聊天领域的集合或单会话 handler，URL 和协议不因内部目录调整而变化。

## 5. 新代码落位

- 新消息类型：稳定结果放入 `features/ai/domain`，渲染器放入 `features/chat/client/renderers`。
- 新模型或 Provider：目录定义放入 AI domain，实例与密钥解析放入 AI server。
- 新 Tool：执行和外部请求放入 `features/ai/server/tools`，公开结果类型放入 AI domain。
- 新持久化方式：实现 `SessionRepository`，根据运行位置放入 chat client 或 server。
- 新基础组件：无业务含义且跨领域复用时才放入 `shared/ui`。
